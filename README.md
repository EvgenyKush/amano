# Квиты — Telegram-бот обмена умениями

Бот для закрытой группы 30–50 человек: учёт сделок, долгов и лимитов без денег. Час — единица учёта, не валюта. Закрыли обмен — вы **квиты**.

## Стек

- Python 3.11+
- aiogram 3.x
- SQLite (aiosqlite)
- APScheduler (напоминания)

---

## 1. Создание бота в BotFather

1. Откройте [@BotFather](https://t.me/BotFather) в Telegram.
2. Отправьте `/newbot`.
3. Введите **имя** бота (например: `Квиты Тест`).
4. Введите **username** (например: `kvity_test_bot`) — должен заканчиваться на `bot`.
5. BotFather пришлёт **токен** вида `7123456789:AAH...` — сохраните его.

### Команды меню (опционально)

BotFather → `/mybots` → ваш бот → **Edit Bot** → **Edit Commands**, вставьте:

```
start - Регистрация
me - Мой профиль
people - Участники
find - Поиск по умениям
deal - Новая сделка
deals - Мои сделки
stats - Сводка (консьерж)
matches - Пересечения (консьерж)
rings - Кольца долгов (консьерж)
export - CSV-выгрузка (консьерж)
broadcast - Рассылка (консьерж)
pause - Пауза участника (консьерж)
unpause - Снять паузу (консьерж)
```

> При запуске `bot.py` команды также выставляются автоматически.

## 2. Локальный запуск

```bash
cd kvity-bot

python -m venv .venv

# Windows
.venv\Scripts\activate

# macOS / Linux
source .venv/bin/activate

pip install -r requirements.txt

cp .env.example .env
# Отредактируйте .env — вставьте BOT_TOKEN и CONCIERGE_IDS

python bot.py
```

### Файл `.env`

```env
BOT_TOKEN=7123456789:AAHxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxx
CONCIERGE_IDS=123456789,987654321
DEBT_LIMIT=2
DEBT_DAYS=14
```

| Переменная | Описание |
|---|---|
| `BOT_TOKEN` | Токен от BotFather |
| `CONCIERGE_IDS` | Telegram ID консьержей через запятую |
| `DEBT_LIMIT` | Макс. открытых долгов на человека (по умолчанию 2) |
| `DEBT_DAYS` | Дней на закрытие долга (по умолчанию 14) |

## 3. Как узнать свой Telegram ID

1. Откройте [@userinfobot](https://t.me/userinfobot).
2. Нажмите **Start**.
3. Бот пришлёт ваш **Id** — число вроде `123456789`.
4. Впишите его в `CONCIERGE_IDS` в `.env`.

## 4. Деплой на Railway

1. Залейте репозиторий на GitHub.
2. Зайдите на [railway.app](https://railway.app) → **New Project** → **Deploy from GitHub repo**.
3. Выберите репозиторий `kvity-bot`.
4. **Variables** → добавьте:
   - `BOT_TOKEN`
   - `CONCIERGE_IDS`
   - `DEBT_LIMIT=2`
   - `DEBT_DAYS=14`
5. **Settings** → **Start Command**: `python bot.py`
6. **Volume** (важно для SQLite):
   - Add Volume → mount path: `/app` (или `/data`)
   - В `config.py` путь к БД — `kvity.db` в корне проекта; при mount на `/app` файл сохранится между перезапусками.
7. Deploy.

> На бесплатном тарифе Railway сервис «засыпает» без трафика. Для 30–50 человек в тесте этого обычно достаточно.

## 5. Чек-лист первого дня теста

- [ ] Все участники прошли `/start` (онбординг: имя, умения, потребности)
- [ ] Консьерж проверил `/stats` — все на месте
- [ ] Консьерж запустил `/matches` — есть пересечения
- [ ] Две пары создали сделки через `/deal` и приняли предложения
- [ ] Одна пара прошла полный цикл: услуга оказана → подтверждение → «Мы квиты» + оценки
- [ ] Проверили лимит: у участника 2 открытых долга → третья сделка блокируется при «Принять»
- [ ] Консьерж сделал `/broadcast` с правилами группы
- [ ] Договорились о канале связи для споров

## 6. Сценарий проверки кольца (/rings)

Нужны **3 Telegram-аккаунта**: A, B, C и консьерж.

### Шаг 1 — регистрация

| Участник | Умею | Нужно |
|---|---|---|
| A | «Помощь с переездом» | «Урок английского» |
| B | «Урок английского» | «Ремонт крана» |
| C | «Ремонт крана» | «Помощь с переездом» |

### Шаг 2 — три сделки (попарно)

1. **A → B**: A даёт переезд (2 ч) ⇄ B даёт английский (2 ч) — принять обеим.
2. **B → C**: B даёт английский (2 ч) ⇄ C даёт ремонт крана (2 ч).
3. **C → A**: C даёт ремонт (2 ч) ⇄ A даёт переезд (2 ч).

### Шаг 3 — оказать только первую услугу в каждой сделке

В каждой сделке исполнитель жмёт **«Я оказал(а) услугу»**, вторая сторона — **«Подтверждаю, услугу получил(а)»**.

В итоге граф долгов:

```
A → B → C → A  (три открытых долга-кольцо)
```

### Шаг 4 — консьерж

```
/rings
```

Бот покажет кольцо `A → B → C → A` с кнопкой **«Предложить зачёт»**.

### Шаг 5 — взаимозачёт

Консьерж жмёт **«Предложить зачёт»** → A, B, C получают карточку → каждый жмёт **«Согласен(на)»**.

После согласия всех троих — три долга закрыты, поздравление.

## Веб-прототип

В папке `web/` лежит самодостаточный HTML-макет интерфейса «Квиты» — без сервера и сборки. Подходит для демо UX, презентаций и согласования экранов до доработки бота.

```bash
# Windows — просто откройте файл в браузере
start web\index.html

# или локальный сервер (удобнее для некоторых браузеров)
cd web
python -m http.server 8080
# → http://localhost:8080
```

**Что можно пройти в демо:**

- Поиск участников, фильтры, текстовый запрос
- Профиль участника и конструктор сделки (равный / неравный обмен с подтверждением)
- Экран «Мой профиль»: умения, долги, пауза / снятие паузы
- «Мои квиты»: кольцо зачёта, матч, черновик сделки → чат
- Чат сделки: карточки, подтверждение услуги, оценка, «мы квиты»
- Групповые проекты (обменный и общественный)
- RU / EN / ES — язык сохраняется в `localStorage`
- Кнопка «сбросить демо» в верхней панели

Лимиты в прототипе совпадают с ботом: **2 долга**, **14 дней** на закрытие, **7 дней** на черновик.

---

## Логика долгов и лимитов

- Лимит **2 открытых долга** проверяется при **принятии** сделки.
- **Просроченный** долг блокирует новые сделки для должника.
- Напоминания: за 3 дня и в день дедлайна (ежедневно в 10:00 UTC).
- Draft-сделки без ответа 7 дней — автоотмена.
